## 3. MVP Convention (Client-Side)

Every gameplay system under `Scripts/MVP/` follows this subfolder layout:

```
<SystemName>/
├── Model/        # Pure data classes, DTOs, domain objects
├── Presenter/    # Business logic — bridges Model ↔ View, no MonoBehaviour
├── Service/      # Stateless utility/helper classes consumed by Presenters
├── SO/           # ScriptableObject definitions (config, catalogs)
└── View/         # MonoBehaviour UI controllers — ONLY talks to its Presenter
```

**Rules:**
- Views NEVER contain business logic. They forward user input to Presenters and render state.
- Presenters are plain C# classes (not MonoBehaviours). They are instantiated by Views.
- Models are pure data containers. No Unity API calls.
- When adding a new system, replicate this exact folder structure.

## 4. Golden Architecture Rules

### 4.1 Multiplayer Authority — HOST IS LAW

| Principle | Detail |
|-----------|--------|
| **Single Source of Truth** | The Photon Room Host holds the authoritative game state in RAM. |
| **Client role** | Clients send **RPCs** to the Host. Clients NEVER write to the database. |
| **Host role** | Receives RPCs → mutates its own in-memory state → sets `isDirty` flags → syncs updated state back to all clients via PUN2 serialization / RPCs. |
| **No client-side DB calls** | `UnityWebRequest` / `HttpClient` calls to the backend are ONLY made by the Host process. |

### 4.2 Delta Batching Save System

| Principle | Detail |
|-----------|--------|
| **No per-action HTTP** | NEVER send an HTTP request for every player action (till, plant, water, harvest). |
| **Dirty tracking** | `ChunkDataSyncManager` marks chunks dirty via `MarkDirty(worldX, worldY)`. `WorldSaveManager` maintains a `HashSet<(chunkX, chunkY, sectionId)>` of dirty chunks. |
| **Batch interval** | The Host auto-saves every **~17 seconds**. `WorldSaveManager.AutoSaveCoroutine()` builds a single delta payload and sends it to `PUT /player-data/world`. |
| **Payload shape** | `{ worldId, day, month, hour, minute, gold, characters[], deltas: ChunkDeltaDto[] }` — only dirty chunks, only non-empty tiles within those chunks. |
| **Retry on failure** | Failed saves keep dirty chunks marked; they are retried on the next cycle. |

### 4.3 Database Integrity (Server)

| Principle | Detail |
|-----------|--------|
| **Atomic saves** | `world.service.ts → saveWorld()` MUST use MongoDB **Transactions** to update World metadata, Characters, Tile deltas, and Inventory simultaneously. |
| **Fallback** | In standalone (non-replica-set) dev environments, fall back to non-transactional writes with a logged warning. |
| **No item duplication** | All inventory mutations happen inside the same transaction as world/tile mutations. |

### 4.4 MongoDB Schema — Tile Storage

| Principle | Detail |
|-----------|--------|
| **No arrays for tiles** | NEVER store chunk tiles in a plain Array. |
| **Use Map (Dictionary)** | Tile data uses `Record<string, TileDataDto>` where the key is the **local tile index** (`"0"` – `"899"` for 30×30). |
| **Why** | Enables O(1) atomic `$set` updates on individual tiles: `tiles.${index}: data`. |
| **Delta writes** | `applyTileDeltas()` uses targeted `$set` operators — it never overwrites unchanged tiles. |

### 4.5 Visual System — Paper Doll / Layered Sprites

| Principle | Detail |
|-----------|--------|
| **No hardcoded sprites** | Sprites are data-driven. `SkinCatalogManager` holds sprite catalogs keyed by `configId`. |
| **DynamicSpriteSwapper** | Runs in `LateUpdate`. Reads the master body renderer's current sprite name → extracts frame index → applies the matching frame from the equipment/tool sprite array. |
| **Sprite download** | Spritesheets are fetched from the server, cached to `Application.persistentDataPath`, and sliced at runtime using a **uniform max-bounding-box grid** (e.g., 64×64 cells). |
| **Adding equipment** | Create a new `configId` entry in the catalog. Never add sprite-selection `if/else` chains. |

### 4.6 Animation Data

| Principle | Detail |
|-----------|--------|
| **Data-driven** | Animation parameters (lock durations, trigger names, frame counts) come from a **server-side AnimationConfig catalog**. |
| **No magic floats** | NEVER hardcode `yield return new WaitForSeconds(0.42f)` for animation timing. Read durations from config. |

### 4.7 Input Handling

| Principle | Detail |
|-----------|--------|
| **Unity New Input System** | All input goes through `InputManager` (singleton) wrapping `FarmittyInputActions`. |
| **Action Maps** | Player actions: Move, Interact, Harvest, Attack, UseItem, ScrollItem, HotbarSlot1–9, OpenInventory, OpenChat, UseSkill. |
| **Rebinding** | Keybind overrides persist via `PlayerPrefs`. `SaveBindings()` / `LoadBindings()` / `ResetBindingsToDefault()`. |
| **Enable/Disable** | Use `InputManager.Instance.DisablePlayerActions()` / `EnablePlayerActions()` when UI panels consume focus. |

## 5. Strict Anti-Patterns — DO NOT DO THESE

### 5.1 Client (Unity / C#)

| ❌ Anti-Pattern | ✅ Correct Approach |
|----------------|---------------------|
| `Input.GetKeyDown(KeyCode.E)` | `InputManager.Instance.Actions.Player.Interact.performed += ...` |
| Business logic inside a `View` MonoBehaviour | Move logic to a `Presenter` class; View only calls Presenter methods |
| `new WWW(...)` or `UnityWebRequest` from a non-Host client | Only the Host makes HTTP calls; clients send RPCs |
| Sending an HTTP request on every tile interaction | Batch via `WorldSaveManager` dirty-chunk system (~17s interval) |
| Storing tile data in `List<TileData>` or arrays on Mongo | Use `Dictionary<string, TileDataDto>` (Map) with string index keys |
| Hardcoding sprite references in `if/else` chains | Use `SkinCatalogManager` + `configId` lookup |
| `yield return new WaitForSeconds(0.5f)` for animation lock | Read duration from AnimationConfig catalog |
| Creating a new enum file for a feature | Add to `Scripts/Enum/GameEnums.cs` — single file for all enums |
| Putting scripts outside `Scripts/MVP/<System>/` without reason | Follow the MVP folder convention |
| `FindObjectOfType<T>()` in `Update()` | Cache references in `Start()` / `Awake()` or inject via inspector |

### 5.2 Server (NestJS / TypeScript)

| ❌ Anti-Pattern | ✅ Correct Approach |
|----------------|---------------------|
| Writing world + inventory in separate non-transactional calls | Use a MongoDB session/transaction in `saveWorld()` |
| Using Mongoose arrays for chunk tiles | Use `Record<string, TileDataDto>` (Mongoose `Map` type) |
| Exposing internal service endpoints without the Gateway | All external traffic goes through `gateway-service` with auth middleware |
| Skipping DTO validation | Use `class-validator` decorators on all DTOs; enable `ValidationPipe` |
| Hardcoding secrets | Use `.env` files per microservice; access via `ConfigService` |

## 6. Data Flow Standard

This is the canonical path for any player action (e.g., "Player plants a seed"):

```
┌─────────────────────────────────────────────────────────────────────┐
│                        CLIENT (Non-Host)                            │
│                                                                     │
│  1. InputManager fires "UseItem" action                             │
│  2. HotbarPresenter.OnItemUsed invoked                              │
│  3. ItemUsageController switches on ItemType                        │
│  4. ItemUsagePresenter.UseSeed() called                             │
│  5. Sends PUN2 RPC to Host ──────────────────────────┐              │
└──────────────────────────────────────────────────────│──────────────┘
                                                       │
┌──────────────────────────────────────────────────────▼──────────────┐
│                        HOST (Room Master)                           │
│                                                                     │
│  6. RPC received → validates action                                 │
│  7. Mutates in-memory state (TileData, Inventory)                   │
│  8. Calls ChunkDataSyncManager.MarkDirty(worldX, worldY)            │
│  9. Syncs updated state to ALL clients via PUN2 RPC / serialization │
│                                                                     │
│  ── Every ~17 seconds (WorldSaveManager) ──                         │
│ 10. BuildPayload() collects only dirty chunks                       │
│ 11. PUT /player-data/world  ─────────────────────────┐              │
└──────────────────────────────────────────────────────│──────────────┘
                                                       │
┌──────────────────────────────────────────────────────▼──────────────┐
│                        SERVER (NestJS)                               │
│                                                                     │
│ 12. gateway-service authenticates JWT                                │
│ 13. player-data-service.saveWorld() opens MongoDB Transaction       │
│ 14. Upserts world metadata (day, month, time, gold)                 │
│ 15. Upserts characters (positions)                                  │
│ 16. applyTileDeltas(): per-tile $set on dirty chunks only           │
│ 17. Applies inventory deltas                                        │
│ 18. COMMIT transaction (all-or-nothing)                             │
└─────────────────────────────────────────────────────────────────────┘
```

**Key invariants along this flow:**
- Steps 1–5: No HTTP call is made. No DB write occurs.
- Steps 6–9: State lives in RAM. `isDirty` flags are set. Clients receive the update in real-time via PUN2.
- Steps 10–11: One batched HTTP request per save cycle. Not per action.
- Steps 12–18: One atomic transaction. Partial failure → full rollback.

## 9. Coding Standards

### C# (Unity)
- Target **.NET Standard 2.1**.
- Use `#region` blocks sparingly — prefer small, focused classes.
- Prefer `[SerializeField] private` over `public` fields on MonoBehaviours.
- Namespaces are NOT used in the current codebase. Do not introduce them unless the team decides to.
- Event naming: `OnSomethingHappened` (e.g., `OnItemUsed`, `OnBreedingResult`).
- Use `static event` for global decoupled events (see `CropBreedingView.OnBreedingResult`).
- Coroutine-based timers are acceptable. `async/await` with UniTask is acceptable where already adopted.

### TypeScript (NestJS)
- **Strict mode** enabled.
- Use `class-validator` + `class-transformer` for all DTOs.
- Controllers are thin — delegate to Services.
- One module per domain entity (`world.module.ts`, `character.module.ts`).
- Database schemas use `@Schema()` / `@Prop()` decorators with Mongoose.
- Environment config via `.env` per microservice + NestJS `ConfigModule`.
